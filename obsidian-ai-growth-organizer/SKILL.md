# Obsidian AI 自成长整理器

把 vault 当作用户拥有的长期数据资产。人的职责是判断、决策和确认；AI 的职责是盘点、整理、关联、检索、生成草稿，并把稳定反馈沉淀为可复核的候选规则。

本技能吸收花叔《Obsidian + Claude Code：用 AI 重建你的第二大脑》的核心思想：本地 Markdown 是 AI 友好的长期载体，规则负责约束行为，Skill 负责封装流程，双向链接负责织网；在此基础上增加安全闸门、渐进读取、批次控制和受控成长机制。

## 输入契约

尽量从当前目录和现有规则推断，不重复询问已有信息：

- `vault`：目标知识库根目录；无法可靠识别时必须询问。
- `goal`：诊断、设计结构、整理指定范围、建立维护流程、创建自成长 Skill，或上述组合。
- `scope`：用户指定的目录、文件或时间范围；未指定时只做结构与规则层盘点，不扫描全部正文。
- `mode`：`audit`、`pilot` 或 `apply`。默认 `audit`；首次实际整理默认 `pilot`。
- `batch_size`：服从库内规范；没有规范时最多 20 篇试点、50 篇单批。

知识库可以是空的，也可以只有零散笔记、部分目录或不完整规则。缺少规范不是失败条件；它决定技能应进入“引导初始化”或“渐进补全”，而不是假定成熟库结构已经存在。

## 不可突破的边界

- 先读取 vault 根目录的 `AGENTS.md`、`CLAUDE.md`、`AI-RUNBOOK.md`、README 或同类规则，再读它们指向的 schema、术语表、目录主页和模板。
- 原始记录、日记、逐字稿、归档与外部来源是源数据。除非规则中存在明确 AI 可写区域，否则不覆盖正文。
- 没有用户明确确认时，不执行批量移动、批量重命名、删除、归档、对外发送、云端上传或敏感数据处理。
- 不猜测负责人、截止日期、事实结论、客户意向、保密级别或实体对应关系；统一标记“待确认”。
- 不创建只有链接没有内容的实体页。找不到实体页时先生成链接建议。
- 不读取被规则排除的目录；常见例子是归档、附件、版本控制目录、Obsidian 插件缓存和私密区。
- 不把公司、客户、健康、投资等内容跨边界混合。外部 AI、embedding 或插件只有在规则允许或用户确认后才能使用。
- “自成长”不得自动修改根规则、活动 Skill 或安全边界。AI 只能生成候选变更；用户确认后才晋升。

## 工作流总览

按 `发现 → 建模 → 预演 → 试点 → 验证 → 扩展 → 学习` 推进。不要一上来扫描或重排整个 vault。

### 1. 识别 vault 与适用规则

1. 确认目标目录确实是 Obsidian vault 或 Markdown 知识库。存在 `.obsidian/` 是强证据，但不是必要条件。
2. 渐进读取：先规则、索引、属性规范、目录主页与摘要，再根据任务读取少量正文。
3. 记录规则优先级：用户本轮要求 > 更接近文件的目录规则 > 根规则 > 本技能默认值。
4. 检查工作区是否已有未提交修改。只处理本次明确范围，保留用户原有变更。
5. 根据实际证据判定库成熟度，不把“缺文件”误判为损坏：

| 状态 | 判断信号 | 默认策略 |
|---|---|---|
| `empty` 空库 | 没有 Markdown 正文，或只有 `.obsidian/` | 引导初始化最小骨架与首个用例 |
| `sparse` 稀疏库 | 只有少量零散笔记，无稳定目录或规则 | 先理解内容与使用目标，再建立轻量规范 |
| `partial` 不完整库 | 已有部分目录、规则、模板或属性，但缺项/互相不一致 | 保留已确认部分，生成缺口清单并渐进补全 |
| `mature` 成熟库 | 有明确规则、路由、模板和稳定内容 | 按现有规则审计、试点和优化 |

无法区分时按风险更低的状态处理，并把判断依据与不确定性写入报告。

### 2. 空库与不完整库引导

#### 空库：从首个真实用例开始

空库没有可供 AI 学习的历史证据，因此不要假装已经理解用户，也不要一次创建庞大的目录、标签和实体体系。

1. 从上下文取得或用最少问题确认：主要用途、常见输入、最重要的输出、是否含敏感内容。用户未回答时可生成“通用最小方案”，但必须标记为待确认。
2. 先输出初始化预演，默认只建议或创建：根规则、收件箱、AI 输出区、模板区、一个入口页，以及库专属 Skill 的成长记录区。
3. 目录名称优先遵循用户语言和现有命名；没有偏好时才采用本技能的最小回退结构。
4. 创建一篇示例/模板和一个真实工作流即可，不创建空白人物、机构、概念或项目页面。
5. 把首批 3–10 篇真实输入作为校准期。校准期只收集观察和候选，不因为单个样本固化分类体系。
6. 每次新增结构都说明用途、触发条件和未来可删除/合并方式，避免空库一开始就过度设计。

#### 不完整库：补缺而不是重建

1. 对目录、规则、schema、模板、索引、AI 写入区、保密边界和 Skill 分别标记：`confirmed`、`inferred`、`missing`、`conflicting`。
2. 已存在且不冲突的内容视为用户资产，默认保留；不要为了统一风格整库重写。
3. `inferred` 项必须给出证据和置信度，只能用于提案或小样本试点，不能当作正式规则。
4. `missing` 项按风险排序补全：先原文保护与保密边界，再写入区域和批量限制，最后才是目录、标签和美化。
5. `conflicting` 项暂停相关写入，展示冲突来源和两个或更多处理选项，等待用户确认。
6. 新规范先放在独立提案或 AI 可写区；只有用户确认后，才合并进根规则或活动 Skill。

### 3. 生成只读盘点

盘点至少覆盖：

- 目录结构与层级深度
- Markdown、Canvas、Base、附件及其他文件类型数量
- frontmatter 覆盖率、常用字段和明显的不一致
- 孤立笔记、失效 wikilink、重复标题与同义词候选
- 收件箱、项目、领域、资源、日志、AI 输出和归档的现有路由
- 已有规则、模板、索引、自动化和 Skill
- 敏感区域、AI 可写区域与明确禁区
- 库成熟度与判断证据
- 关键能力缺口：规则、schema、模板、索引、写入区、保密边界和 Skill

不要为了统计而读取所有正文。先用文件名、路径、属性头和链接做结构分析；只有判断分类质量或抽样验证时才读取正文。

盘点输出使用[诊断与变更报告模板](references/report-templates.md)。结论必须附文件路径或 `[[来源笔记]]`。

### 4. 建立最小可用架构

优先适配现有系统，不强迫用户换成固定的 PARA、LYT 或其他流派。没有任何规范时，才建议最小结构：

```text
00-Inbox/        原始输入与待处理内容
01-Dashboard/    入口、状态和待确认事项
02-Projects/     有明确结果和结束条件的工作
03-Areas/        长期责任
04-Resources/    可复用知识
05-Logs/         日记与复盘
07-AI-Outputs/   可重新生成的 AI 报告和建议
90-Templates/    模板
99-Archive/      默认不读取
```

最小规范应定义：必填属性、类型与状态枚举、命名、链接规则、AI 写入边界、保密分级、批量上限和完成标准。把针对具体库的规则写进该 vault 的规则文件，不把用户名、公司名、绝对路径或业务分类硬编码进本技能。

对空库，第一版规范只保留立即需要的字段和工作流；对不完整库，以“最小补丁”补齐高风险缺口。不要因为本技能提供了完整清单，就要求一次性全部实现。

### 5. 先输出整理蓝图和预演

实际修改前给出：

- 目标与非目标
- 候选范围、文件数量和抽样方法
- 建议目录/属性/命名/链接映射
- 将新增、修改、移动、跳过的数量
- 风险、冲突与待确认项
- 回退方式；有 Git 时建议用差异审阅，不自行提交

`audit` 模式到此停止。`pilot` 模式只处理小样本。`apply` 也必须先经过预演，除非用户已经明确批准同一份变更清单。

### 6. 执行小批次整理

每篇笔记按以下顺序处理：

1. 判断它是源数据还是 AI 衍生内容，确定可写范围。
2. 保留原文；需要摘要时写入规则指定的 AI 区域，或另建 AI 衍生文件。
3. 补属性时只填有证据的字段；不确定值写“待确认”或省略可选字段。
4. 先查词汇表、别名与已有标题，再建议 1–5 个直接相关的 wikilink。
5. 优先增量更新已有知识条目；只有内容足以独立成立时才建议新建。
6. 记录来源、执行时间、处理动作和跳过原因。
7. 一批结束立即校验，不跨批积累错误。

### 7. 验证并交付

检查：YAML 可解析、必填属性齐全、wikilink 语法未损坏、原文未覆盖、路径符合规则、AI 内容可识别、数量与报告一致。

最终报告必须区分：已完成、仅建议、已跳过、失败、待确认。若用户还未批准写回，清楚说明“当前没有修改源笔记”。

## AI 自成长闭环

成长的对象是“库专属操作经验”，不是未经监督地扩张权限。为目标 vault 建议以下资产；路径优先服从现有规范：

```text
knowledge-rules/
  AGENT-RULES.md             稳定规则与路由
  skills/<vault-skill>/      当前生效的库专属 Skill
  ai-learning/
    observations.md          只追加的观察与纠错证据
    candidates.md            待验证候选规则
    changelog.md             已批准晋升记录
    evals.json               回归测试提示
```

### 捕获学习信号

仅从可追溯事件中学习：用户明确纠正、同一手工步骤反复出现、同类错误重复发生、规则冲突、或用户明确认可的高质量结果。每条观察记录日期、任务、证据路径、原行为、用户反馈、候选改进和保密边界。不要从沉默推断偏好。

### 形成候选规则

满足以下任一条件才创建候选：

- 同类纠正至少出现 2 次；
- 用户明确说“以后都这样”；
- 现有规则与实际目录产生可复现冲突；
- 一个重复流程已经有稳定输入、步骤、输出和完成标准。

候选必须标注适用范围、反例、风险、证据和建议测试。一次性偏好、具体客户事实、临时路径、密钥和个人敏感内容不得写入通用 Skill。

### 验证与晋升

1. 为候选增加至少一个正向用例、一个边界用例和一个不应触发的近似用例。
2. 在副本、临时目录或 3–5 篇小样本上运行；比较是否减少错误且不破坏原规则。
3. 输出 Skill 差异、测试结果和影响范围。
4. 只有用户明确确认后，才把候选写入活动 Skill 或规则。
5. 每次晋升写 changelog，并保留回退点。安全规则、保密边界和删除权限不能通过学习机制放宽。

学习记录与候选格式见[成长记录模板](references/growth-templates.md)。

## 库专属 Skill 的生成要求

当用户要求“给这个库生成 Skill”时，创建 `<vault-skill>/SKILL.md`，至少包含：

- 强触发描述：整理、同步、检索、复盘等真实用户表达
- vault 定位方法，不写死制作者机器路径
- 规则渐进读取顺序
- 目录路由、属性 schema、模板与术语表引用
- 源数据、AI 区域、保密和批量操作边界
- 3–7 个高频工作流
- 每个工作流的输入、范围、输出、验证和报告
- 自成长观察、候选、测试、人工晋升与回退机制
- `evals/evals.json` 中至少 3 个真实测试：安全整理、边界冲突、自成长候选

保持主 `SKILL.md` 精简；详细 schema、模板和领域差异放进 `references/`，确定性重复操作才放进 `scripts/`。

## 完成标准

- 所有结论能追溯到规则、源笔记或本次统计。
- 未经批准没有批量移动、删除、覆盖或外发。
- 结构建议适配用户现有系统，而不是套用固定流派。
- 空库从最小骨架和真实用例启动，没有批量生成空白页面或过度设计。
- 不完整库有 `confirmed/inferred/missing/conflicting` 缺口清单，补全没有覆盖已确认资产。
- 源数据与 AI 衍生内容可区分，敏感边界未混合。
- 实际变更有差异、数量、验证结果和待确认清单。
- 自成长资产有证据、测试、人工确认和回退，不会自行放宽权限。
