# 关闭已确认输入

把“发现归档候选”“人工批准”和“关闭来源”分为三个阶段。本技能可以生成未批准候选，也可以执行已经批准的迁移；不得把候选发现等同于归档授权。

## 开始前读取

依次读取：

1. 根目录 `AGENTS.md` 和 `AI-RUNBOOK.md`。
2. `知识库规则/SCHEMA.md`、`00-收件箱/收件箱说明.md`。
3. `知识库规则/自动化/已确认输入归档配置.md`。
4. `01-驾驶舱/归档审批.md`。
5. 获批条目的属性、AI处理区、相关衍生结果、目标目录说明和来源同步索引。
6. 处理外部来源时读取[来源适配规则](references/source-adapters.md)，只加载本批涉及的来源段落。

默认不扫描 `99-归档/` 正文全集。只读取目标冲突检查所需的文件名、属性或明确目标。

## 运行模式

- 默认`discover_and_execute`：先刷新待批准候选，再执行已批准条目。
- `discover_only`：只生成或补充待批准候选，不移动文件。
- `execute_only`：只预检和迁移已批准条目，不发现新候选。

定时任务默认使用`discover_and_execute`。发现阶段与执行阶段分别统计，任一阶段零项都要如实报告。

## 发现归档候选

按以下顺序渐进读取，不扫描整个收件箱正文：

1. 各来源同步索引中的本地路径、来源ID、本地状态和处理状态。
2. 最近的每日整理、分流、来源包审计和知识提炼报告。
3. 仅对候选来源读取frontmatter、摘要与AI受控区；需要确认目标建议时才读取相关项目或领域主页。

满足以下条件的来源可进入“待批准”，但不代表已经满足执行门槛：

- 来源仍位于`00-收件箱/`，且未被标记为missing、deleted、moved或已关闭。
- 已经过对应日处理，或存在明确处理记录；纯同步但尚未整理的来源不生成候选。
- 不存在相同`closure_id`的待批准、已批准或已关闭记录。
- 能识别稳定来源ID和来源粒度；飞书按完整来源包生成一条候选，浮墨按单条memo生成。

使用`closure_id: close-<source>-<稳定来源ID>`。将新候选追加到审批页的`ARCHIVE_APPROVAL_CANDIDATES`受控区，格式为：

```markdown
- [ ] #归档批准 [[来源笔记或来源包主笔记]] <!-- closure_id: close-来源-稳定ID -->
  - 来源类型：flomo | feishu | xiaohongshu | douyin | obsidian-direct | other
  - 处理方式：建议值 / 待确认
  - 目标位置：[[建议唯一目标]] / 待确认
  - 衍生结果：[[相关卡片或报告]] / 不适用 / Not applicable
  - 审批状态：pending / 待批准
  - 待确认：具体缺口 / 无 / None
  - 索引：[[来源同步索引]] / 不适用 / Not applicable
  - 候选生成时间：YYYY-MM-DDTHH:mm:ss+时区
  - 确认时间：
```

发现阶段只能追加新`closure_id`或补充完全空白的建议字段。不得勾选候选，不得把`pending`改为`approved`，不得覆盖用户修改的目标、处理方式、待确认、确认时间或复选框状态。来源不再符合候选条件时保留记录并追加`候选状态：stale / 已失效`，不静默删除。

## 唯一授权入口

执行阶段只处理 `01-驾驶舱/归档审批.md` 中符合以下格式且已经勾选的条目：

```markdown
- [x] #归档批准 [[来源笔记]] <!-- closure_id: close-来源ID -->
  - 处理方式：项目材料 | 内容素材 | 日志 | 保留备查 | 知识提炼完成
  - 目标位置：[[唯一目标目录或文件]]
  - 衍生结果：[[相关卡片或报告]]
  - 审批状态：approved / 已批准
  - 确认时间：YYYY-MM-DD
```

复选框勾选表示用户批准移动这一条来源记录到列明的目标，不代表批准删除。仅在对话中说“整理完成”、来源状态变为 processed、候选任务已转换、文章卡已生成或超过一定天数，都不构成归档授权。

用户批准时必须同时勾选复选框、把审批状态改为`approved / 已批准`并填写确认时间。只有其中一项不算完整授权。

## 全局统一审批门槛

飞书、浮墨、小红书、抖音、Obsidian直写、通话记录和附件统一使用同一组授权条件，不得由来源技能另设更宽松入口：

1. 审批行已勾选并包含`#归档批准`。
2. `closure_id`稳定且唯一。
3. 来源链接可解析，来源文件或来源记录存在于收件箱。
4. 处理方式明确，目标位置唯一且位于知识库内。
5. `审批状态：approved / 已批准`，确认时间存在。
6. 影响归属的待确认项为“无 / None”。
7. 必要衍生结果已完成，或明确写“不适用 / Not applicable”。
8. 目标无路径、同名或来源ID冲突。
9. 来源索引或台账可安全更新；无索引来源明确写“不适用 / Not applicable”。
10. 不涉及删除、覆盖或扩大到审批行之外的文件。

以上十项是所有来源相同的审批门槛。来源适配器只能增加完整性和索引写法检查，不能减少或替换这十项。

## 预检

逐条验证，任一条件失败就跳过并报告：

- `closure_id` 存在且在清单、报告和来源索引中唯一。
- 来源位于 `00-收件箱/`，文件存在且未被其他流程移动。
- 处理方式和目标位置明确；目标位于知识库内且符合目录路由。
- AI分类或归属建议已经人工确认，不再含影响路由的“待确认”。
- 要求的闪念卡、文章卡、正式任务、知识笔记或处理报告已经存在；不适用时清单明确写“无”。
- 来源笔记没有尚未处置的候选待办；已批准候选必须已成功转换，未批准候选必须保留在随来源一起迁移后仍可查询的位置，或由用户明确决定继续保留。
- 来源索引可识别且能安全更新；飞书来源包必须完整并整体迁移。
- 目标不存在同名文件、相同来源 ID 或其他冲突。

先生成预检结果。单批遵守配置上限；超过上限时分批，不扩大授权范围。

## 路由

遵守收件箱说明中的路由：

- 明确属于项目：迁入对应项目的 `材料/` 或 `会议/`。
- 内容素材：迁入 `06-素材库/` 对应主题。
- 日志记录：迁入 `05-日志/` 对应目录。
- 已完成知识提炼、仅作证据或保留备查：迁入 `99-归档/来源名称/` 下对应目录。

“删除候选”不在无人值守归档范围。删除必须由用户在当前任务中再次明确授权，并按知识库删除规则执行。

## 执行顺序

1. 为本批记录来源路径、目标路径、来源 ID、索引位置和相关双链。
2. 移动来源文件；成组来源一次整体移动。不得覆盖目标文件。
3. 更新迁移后文件的 `status`、`updated` 和适用的关闭信息；保留原文与人工区域。
4. 按[来源适配规则](references/source-adapters.md)更新来源同步索引中的本地相对路径，并记录 `local_state: moved`、`local_state_label: 已迁移 / Moved`、`moved_at`、`destination_type`和`destination_type_label`。索引不删除历史来源 ID。浮墨同步技能不再接受迁移清单，本步骤只能由本技能执行。
5. 修复本次迁移直接造成的双链失效；不顺便重写无关笔记。
6. 验证来源已不在收件箱、目标文件存在、索引指向目标且衍生结果仍可追溯。
7. 在审批行追加 `#已归档`、`closed_to`、`closed_at`和`关闭状态：closed / 已关闭`。只有全部验证成功后才写成功标记。
8. 生成 `07-AI产出/整理报告/YYYY-MM-DD-已确认输入关闭报告.md`，列出成功、跳过、失败、原路径、目标路径和索引更新结果。

报告还要列出发现阶段的扫描索引数、处理记录数、新增候选数、重复跳过数、失效候选数和仍缺少目标的候选数。

部分失败时保留可恢复证据，不把审批项标成成功；能够安全回退本次单条移动时回退，不能安全回退时立即停止该条并报告人工处理步骤。

## 幂等与完成检查

- 已含 `#已归档`、`closed_to` 和 `closed_at` 的审批项只审计，不重复移动。
- 来源、目标和索引三处状态一致；文件不再残留在收件箱。
- 原始内容未改写，来源 ID 和历史索引未丢失。
- 飞书来源包未拆分；浮墨单条 memo 仍对应唯一 Markdown。
- 未勾选、目标不唯一、仍待确认或删除候选均未处理。
- 报告数量与实际移动数量一致。
- 新发现候选真实出现在审批页，状态为`pending / 待批准`且未勾选；重复运行不产生相同`closure_id`。
- 发现阶段没有改变任何人工审批字段或触发迁移。
