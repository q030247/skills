# 抖音收藏列表同步

把抖音收藏视为个人源数据，只在用户授权的本地知识库中处理。先判断智能体是否真的具备浏览器能力与有效登录态，再选择自动采集或人工导出路径；工具能力和登录结果都必须有真实证据。

## 输入与输出契约

从用户输入或当前项目规则取得：

- `knowledge_base`：目标知识库；能从项目规则可靠识别时可省略。
- `collection`：可选，指定收藏合集；未指定时同步当前打开或用户选择的合集。
- `max_items`：可选，单次最大条数；服从知识库批次规则，没有规则时最多 50 条。
- `restore_missing`：可选，默认 `false`，不自动恢复用户已在本地标记删除的条目。

最终收藏内容必须写入 Markdown 文件，并以表格为主体。不得只返回 JSON、CSV、截图或聊天中的临时列表。长字段和完整媒体地址放进折叠详情表，主表保持可读。

## 安全边界

- 只读取当前用户自己的抖音收藏，不绕过登录、验证码、风控或访问控制。
- 不提取、展示或保存 Cookie、sessionid、ttwid、Authorization、二维码内容等凭据。浏览器会话已能携带 httpOnly Cookie，无需读取它们。
- 不把收藏、客户、个人偏好或经营信息发送到其他服务。
- 只新增、去重和维护同步状态；不删除、移动、覆盖原始记录，不自动重新同步本地已删除条目。
- 不直接用自造 `fetch()` 请求收藏合集 API。此路径容易被安全层拦截；让抖音页面发出原生请求并观察响应。
- 遇到验证码、异常访问、频率限制或登录失效时立即停止，不反复重试。

## 1. 定位知识库并读取规则

1. 按“用户明确路径 → 项目规则声明 → 当前目录被明确称为 vault/知识库”的顺序定位目标。
2. 读取根目录 Agent 指令、运行手册、schema、目标收件箱说明和既有抖音同步索引；文件名可能不同，应按内容和项目指引识别，不要假定某台机器上的固定路径。
3. 提取目标目录、必填属性、来源字段、AI 可写边界、索引位置、报告位置、批次上限、归档排除范围和保密边界。
4. 没有可靠知识库路径时暂停，让用户提供路径或确认使用当前目录；确认前不访问抖音。
5. 冲突优先级：用户本轮明确要求 > 目标目录规则 > 根目录规则 > 本技能默认值。

将发现结果整理成执行参数表后再采集，至少包括：`inbox_dir`、`douyin_dir`、`index_path`、`report_dir`、`required_properties`、`ai_write_boundary`、`batch_limit`、`excluded_dirs`。这些值来自目标知识库，不能把开发者的目录、用户名、设备名或 vault 名称写进技能或生成物。

具体适配方法见 [知识库适配规则](references/knowledge-base-adaptation.md)。

## 2. 探测浏览器能力

检查当前智能体实际暴露的工具，不根据产品名称猜测。

| 能力状态 | 判定证据 | 后续路径 |
|---|---|---|
| 完整内置浏览器 | 能打开页面、保留会话、检查 DOM，并在页面上下文执行 JavaScript 或观察网络响应 | 进入自动采集 |
| 仅可导航/截图 | 能打开页面但不能执行脚本、读取 DOM 或响应体 | 不能自动全量采集；走人工导出 |
| 无浏览器 | 没有可用浏览器工具 | 走人工导出 |

若只有普通网页搜索/抓取工具，它不等于用户的持久化浏览器会话，也不能证明能读取私人收藏。

### 人工导出路径

无完整内置浏览器时：

1. 明确说明当前能力缺口，不能声称已同步。
2. 给用户一段仅在自己已登录抖音页面中运行的导出脚本或操作说明；脚本只拦截 `listcollection` 响应、自动滚动并下载 JSON，不读取 Cookie。
3. 让用户把导出的 JSON 文件放进授权工作区或上传给智能体。
4. 收到文件后先校验结构，再从“合并去重”继续。

具体的能力分流和人工导出要求见 [浏览器路径](references/browser-paths.md)。

## 3. 判断登录态

自动采集前打开 `https://www.douyin.com/`，使用页面可见状态和最小只读检查验证登录：

1. 页面出现当前用户头像、昵称、个人中心或收藏入口，并且进入收藏页不会跳转登录，视为强证据。
2. 能读取页面公开给前端的当前用户信息（如 `nickname`、`uid`、`sec_uid`）可作为辅助证据；不要输出完整标识。
3. 出现“登录/扫码登录”、登录弹窗、收藏入口不可用或被重定向，视为未登录。
4. 页面状态不明确时，不能根据残留 localStorage 或非 httpOnly Cookie 猜测已登录。

未登录时打开登录页，让用户亲自扫码或完成验证，然后暂停。用户确认后重新检查页面状态；不要代替用户登录。内置浏览器通常会持久化完整会话，无需手动提取 sessionid。

## 4. 采集收藏合集

登录有效后进入用户的收藏合集页面，并按以下顺序执行：

1. 在触发滚动前注入观察器，包装 `window.fetch` 与 `XMLHttpRequest`，只捕获 URL 含 `/aweme/v1/web/aweme/listcollection/` 的成功响应 JSON。保留原函数行为，不阻断页面请求。
2. 不要混用 `/aweme/v1/web/aweme/favorite/`；它是“收藏视频”接口，不是“收藏合集”列表。
3. 找到实际可滚动容器，优先检查 `.route-scroll-container`；若页面结构已变化，通过滚动高度和滚动事件确认容器，不盲目使用旧选择器。
4. 以温和间隔滚动，让页面自身触发分页请求。每轮检查新增响应、游标、`has_more` 和滚动高度。
5. 在 `has_more` 明确为 false，或连续多轮无新增且已到达底部时停止。设置合理的总时限；不要无限滚动。
6. 记录捕获批次数、页面声称的分页结束状态和异常，不记录请求头或 Cookie。

页面改版、响应结构变化或捕获结果为零时停止并报告，不使用未经验证的新接口猜测修复。

## 5. 合并、标准化与去重

1. 从所有捕获响应中合并 `aweme_list`。
2. 使用 `aweme_id` 作为唯一键；同一 ID 多次出现时保留字段更完整或来源更新时间更新的一条。
3. 至少保留：作品 ID、标题/描述原文、作者、发布时间、时长、作品类型、点赞、评论、分享、收藏、话题、封面、分享链接、站内链接、采集时间和收藏合集。
4. 可额外保留作者 UID/sec_uid、尺寸、编码格式、播放进度和 H264/H265 地址；这些字段放进详情表，避免主表过宽。
5. API 未返回播放量时写 `—`，不要把默认数值 0 表述成真实播放量。
6. 不根据封面或链接推测标题、合集、作者或“为什么收藏”。无法确认的内容写“待确认”。
7. 若目标知识库要求记录“为什么收藏”或“支持什么决策/输出”，在表格中保留这两列；同步阶段不替用户补造，默认写“待确认”，供后续整理。

字段映射和表格结构见 [Markdown 输出模板](references/markdown-output.md)。

## 6. 读取索引并计算增量

1. 优先使用知识库规则指定的抖音同步索引；其次在目标目录寻找正文明确声明为抖音索引的 Markdown。
2. 索引唯一键为 `aweme_id`，并保存来源 URL、收藏合集、本地表格路径、首次/最近同步时间和状态。
3. 索引重复 ID、路径冲突或机器记录与正文表格不一致时停止写入，列为待确认。
4. 新 ID 标记 `added`；已登记且内容指纹不变标记 `skipped`；来源字段变化标记 `updated`，但只更新 AI 可维护的表格行。
5. 本地标记删除的 ID 保持删除状态并跳过，除非 `restore_missing: true`。
6. 本轮没看到旧 ID 不能证明来源已删除；只有确定性的全量遍历完成后才能标记 `source_state: missing`，仍不得物理删除本地内容。

## 7. 写入 Markdown 表格

按知识库规范决定路径。若规范把外部来源统一路由到收件箱，并为抖音定义了来源子目录，则写入该子目录；无规范时使用知识库下的 `douyin-favorites/`。技能中不得硬编码创建者本机的绝对路径、用户名、设备名或 vault 名称。

1. 一个收藏合集对应一个 Markdown 列表文件；未分合集时使用 `抖音收藏列表.md`。
2. 文件包含合规 YAML、同步摘要、主表、可选详情表和待确认项。主表按收藏采集顺序或发布时间倒序稳定排序，并在报告注明排序依据。
3. 标题或描述中的换行替换为空格，竖线转义为 `\|`；链接使用 Markdown 链接，数组用顿号或 `<br>`，避免破坏表格。
4. 表格超过知识库单批上限时分批写入同一文件的多个表格区段或多个分页文件，并在索引中记录全部路径。
5. 只修改明确的 AI 区域；既有文件没有 AI 标记时，另建新的同步列表文件，不覆盖人工内容。
6. 文件和索引均写入成功后才把条目标记为同步成功。
7. 同步只负责采集和入箱，不自动把条目移动到项目、领域、资源、素材或归档目录；需要分类时另行生成建议，等待人工确认。

## 8. 生成执行报告

每次执行都生成本地 Markdown 报告。报告位置优先使用知识库为 AI 衍生内容指定的目录；若未定义，则放在抖音同步目标目录的 `reports/` 子目录。报告至少包括：

| 项目 | 内容 |
|---|---|
| 环境 | 浏览器能力与登录验证结果 |
| 范围 | 收藏合集、时间范围、分页完成状态 |
| 统计 | 捕获批次、原始条数、去重后条数、新增、更新、跳过、失败、待确认 |
| 输出 | 收藏表格、索引和报告的本地链接 |
| 风险 | 验证码、页面改版、未覆盖范围和字段缺失 |
| 安全检查 | 是否读取凭据、覆盖人工内容、删除/移动文件或向外部服务发送数据 |

报告本身也以 Markdown 表格呈现关键结果。失败项不能登记为成功。

## 完成检查

- 浏览器能力来自实际工具证据，并正确选择自动或人工路径。
- 登录态来自当前页面证据；未登录时已暂停让用户操作。
- 使用页面原生 `listcollection` 请求，不直接伪造 API 调用。
- 已遍历分页或明确记录未覆盖范围，`aweme_id` 去重无冲突。
- 最终内容是本地 Markdown 表格，字段转义正确，可追溯到作品链接。
- 索引、表格和报告数量一致；原始记录与人工内容未被覆盖。
- Cookie、会话标识和其他凭据未被输出或保存。
