# 抖音收藏列表同步

把抖音收藏视为个人源数据，只在用户授权的本地知识库中处理。先判断智能体是否真的具备浏览器能力与有效登录态，再选择自动采集或人工导出路径；工具能力和登录结果都必须有真实证据。

## 简短调用提示词

用户可以直接复制以下提示词调用本技能：

> 使用 `douyin-favorites-sync` 技能增量同步当前账号的抖音收藏合集。
>
> 先读取当前知识库规则并确定收件箱、同步索引和报告位置。必须复用技能内置的 `douyin_capture.js` 和 `process_favorites.py`，在进入收藏页面前安装拦截器，确保捕获首屏和后续分页数据。
>
> 按 `aweme_id` 增量去重，保留分享链接、视频播放地址、H264/H265 地址及完整查询参数，最终生成 Markdown 收藏表、链接与媒体地址表、同步索引和执行报告。
>
> 同步结束后对账页面显示总数、当前捕获数、索引原有数和合并唯一数。数量不一致时标记部分完成，不得声称全量成功或承诺重跑一定补齐。不得读取凭据、覆盖人工内容、自动移动或删除文件。

## 内置脚本

本技能已经固化重复且易出错的处理逻辑，执行时优先复用，不要临时重写另一套拦截、字段映射、去重或 Markdown 生成代码：

| 脚本 | 职责 | 运行位置 |
|---|---|---|
| `scripts/douyin_capture.js` | 预安装 fetch/XHR 拦截器、合并首屏结构化数据、滚动加载、按 `aweme_id` 预去重并导出 JSON | 用户已登录的抖音浏览器页面 |
| `scripts/process_favorites.py` | 读取一个或多个采集 JSON，稳定提取字段、增量去重、维护 Markdown 索引、生成收藏表格和执行报告 | 用户授权的本地工作区 |

只有页面或 API 结构已经变化、现有脚本明确报错时，才诊断并修改脚本；不要在每次同步时实时生成替代脚本。具体调用方法见 [脚本使用说明](references/script-usage.md)。

## 输入与输出契约

从用户输入或当前项目规则取得：

- `knowledge_base`：目标知识库；能从项目规则可靠识别时可省略。
- `collection`：可选，指定收藏合集；未指定时同步当前打开或用户选择的合集。
- `max_items`：可选，单次最大条数；服从知识库批次规则，没有规则时最多 50 条。
- `restore_missing`：可选，默认 `false`，不自动恢复用户已在本地标记删除的条目。

最终收藏内容必须写入 Markdown 文件，并以表格为主体。不得只返回 JSON、CSV、截图或聊天中的临时列表。长字段和完整媒体地址放进折叠详情表，主表保持可读。

## 安全边界

- 只读取当前用户自己的抖音收藏，不绕过登录、验证码、风控或访问控制。
- 不提取、展示或保存 Cookie、sessionid、ttwid、Authorization、二维码内容等凭据。浏览器会话已能携带 httpOnly Cookie，无需读取它们。
- 不把收藏、客户、个人偏好或经营信息发送到其他服务。
- 只新增、去重和维护同步状态；不删除、移动、覆盖原始记录，不自动重新同步本地已删除条目。
- 不直接用自造 `fetch()` 请求收藏合集 API。此路径容易被安全层拦截；让抖音页面发出原生请求并观察响应。
- 遇到验证码、异常访问、频率限制或登录失效时立即停止，不反复重试。

## 1. 定位知识库并读取规则

1. 按“用户明确路径 → 项目规则声明 → 当前目录被明确称为 vault/知识库”的顺序定位目标。
2. 读取根目录 Agent 指令、运行手册、schema、目标收件箱说明和既有抖音同步索引；文件名可能不同，应按内容和项目指引识别，不要假定某台机器上的固定路径。
3. 提取目标目录、必填属性、来源字段、AI 可写边界、索引位置、报告位置、批次上限、归档排除范围和保密边界。
4. 没有可靠知识库路径时暂停，让用户提供路径或确认使用当前目录；确认前不访问抖音。
5. 冲突优先级：用户本轮明确要求 > 目标目录规则 > 根目录规则 > 本技能默认值。

将发现结果整理成执行参数表后再采集，至少包括：`inbox_dir`、`douyin_dir`、`index_path`、`report_dir`、`required_properties`、`ai_write_boundary`、`batch_limit`、`excluded_dirs`。这些值来自目标知识库，不能把开发者的目录、用户名、设备名或 vault 名称写进技能或生成物。

具体适配方法见 [知识库适配规则](references/knowledge-base-adaptation.md)。

## 2. 探测浏览器能力

检查当前智能体实际暴露的工具，不根据产品名称猜测。

| 能力状态 | 判定证据 | 后续路径 |
|---|---|---|
| 完整内置浏览器 | 能打开页面、保留会话、检查 DOM，并在页面上下文执行 JavaScript 或观察网络响应 | 进入自动采集 |
| 仅可导航/截图 | 能打开页面但不能执行脚本、读取 DOM 或响应体 | 不能自动全量采集；走人工导出 |
| 无浏览器 | 没有可用浏览器工具 | 走人工导出 |

若只有普通网页搜索/抓取工具，它不等于用户的持久化浏览器会话，也不能证明能读取私人收藏。

### 人工导出路径

无完整内置浏览器时：

1. 明确说明当前能力缺口，不能声称已同步。
2. 使用技能内置的 `scripts/douyin_capture.js`，不要现场生成另一段导出脚本。指导用户仅在自己已登录抖音页面中运行；脚本只拦截 `listcollection` 响应、自动滚动并下载 JSON，不读取 Cookie。
3. 让用户把导出的 JSON 文件放进授权工作区或上传给智能体。
4. 收到文件后先校验结构，再从“合并去重”继续。

具体的能力分流和人工导出要求见 [浏览器路径](references/browser-paths.md)。

## 3. 判断登录态

自动采集前打开 `https://www.douyin.com/`，使用页面可见状态和最小只读检查验证登录：

1. 页面出现当前用户头像、昵称、个人中心或收藏入口，并且进入收藏页不会跳转登录，视为强证据。
2. 能读取页面公开给前端的当前用户信息（如 `nickname`、`uid`、`sec_uid`）可作为辅助证据；不要输出完整标识。
3. 出现“登录/扫码登录”、登录弹窗、收藏入口不可用或被重定向，视为未登录。
4. 页面状态不明确时，不能根据残留 localStorage 或非 httpOnly Cookie 猜测已登录。

未登录时打开登录页，让用户亲自扫码或完成验证，然后暂停。用户确认后重新检查页面状态；不要代替用户登录。内置浏览器通常会持久化完整会话，无需手动提取 sessionid。

## 4. 采集收藏合集

登录有效后按以下顺序执行。关键原则是让观察器先于收藏页首屏请求生效；“页面已经打开后、开始滚动前安装”仍然太晚，会漏掉首屏数据。

1. 读取当前索引中的既有 `aweme_id` 集合，用于本轮合并和完整性对账，但不要用索引替代来源采集。
2. 读取内置 `scripts/douyin_capture.js` 的完整文本，在进入或重新加载收藏页之前，优先通过浏览器级 `addInitScript` 或 `evaluateOnNewDocument` 注册；不要复制后手写一个近似版本。脚本会包装 `window.fetch` 与 `XMLHttpRequest`，只捕获 URL 含 `/aweme/v1/web/aweme/listcollection/` 的成功响应 JSON，保留原函数行为，不阻断页面请求。
3. 若收藏页已完成首屏加载：先注册可跨导航生效的初始化脚本，再刷新收藏页；不能只在当前页面 `evaluate` 后立即刷新，因为普通页面脚本会随刷新丢失。没有浏览器级初始化能力时，回到仍保持同一文档的上一级页面安装观察器，再通过站内 SPA 操作进入收藏页，并验证包装函数没有因整页导航失效。
4. 观察器生效后再进入或刷新目标收藏页。立即检查捕获计数，确认首个 `listcollection` 响应已被记录；如果页面已经显示大量收藏但捕获仍为零，停止滚动并修正安装时机。
5. 调用 `window.__douyinFavoritesCapture.collectBootstrap()` 检查页面内已注入的首屏 JSON。内置脚本只合并其中结构明确、含稳定 `aweme_id` 的 `aweme_list` 等价数据；不要根据 DOM 卡片文本拼造完整作品记录。此路径是首屏兜底，不替代预注入观察器。
6. 不要混用 `/aweme/v1/web/aweme/favorite/`；它是“收藏视频”接口，不是“收藏合集”列表。
7. 找到实际可滚动容器，优先检查 `.route-scroll-container`；若页面结构已变化，通过滚动高度和滚动事件确认容器，不盲目使用旧选择器。
8. 调用 `await window.__douyinFavoritesCapture.scroll()` 以温和间隔滚动，让页面自身触发分页请求；用 `.status()` 检查新增响应、游标、`has_more`、滚动高度和唯一作品数。
9. 在 `has_more` 明确为 false，或连续多轮无新增且已到达底部时停止。设置合理的总时限；不要无限滚动。
10. 调用 `.exportData({displayed_total, collection, page_complete})` 取得 JSON，或调用 `.download(filename, meta)` 下载。记录首屏响应是否捕获、捕获批次数、页面声称的分页结束状态、页面显示收藏数、当前运行唯一 ID 数、与索引合并后的唯一 ID 数和异常，不记录请求头或 Cookie。

### 完整性对账

采集结束后必须对账，不能只看到 `has_more: false` 就声称全量完成：

1. 从页面可见统计中读取收藏总数，记为 `displayed_total`；读不到时写“待确认”。
2. 分别计算当前运行捕获的 `captured_unique`、索引已有的 `indexed_before`、两者按 `aweme_id` 合并后的 `union_unique` 和本次新增 `new_unique`。
3. 若 `union_unique == displayed_total`，且分页明确结束，可标记 `completeness: complete`。
4. 若 `union_unique < displayed_total`，标记 `completeness: partial`，记录差额并优先判断是否遗漏首屏；不得更新“全量同步成功”时间。
5. 若 `union_unique > displayed_total`，可能是页面数量变化、合集范围不一致、索引含历史失效作品或页面统计口径不同，标记“待确认”，不能删除索引历史记录。
6. 重新同步只有在新一轮实际捕获到遗漏 ID 时才能补齐。索引会跳过已采集 ID，但不会自动创造仍未被观察器捕获的首屏数据；禁止承诺“重跑必然补齐”。
7. 页面总数可能包含已删除、私密或暂不可访问作品。若完成预注入、首屏兜底和分页遍历后仍有差额，报告具体证据和可能口径差异，不猜测是哪一条。

页面改版、响应结构变化或捕获结果为零时停止并报告，不使用未经验证的新接口猜测修复。

## 5. 合并、标准化与去重

调用内置 `scripts/process_favorites.py` 完成本节到执行报告的确定性处理，不要人工拼接 Markdown 表格。

1. 脚本从所有输入 JSON 和嵌套响应中合并 `aweme_list`。
2. 脚本使用 `aweme_id` 作为唯一键；同一 ID 多次出现时合并非空字段。
3. `视频位置`优先提取作品的 POI 名称、地址或城市信息，其次使用响应中的 IP 归属地；来源没有位置字段时输出 `—`，不根据作者或标题推测。
3. 至少保留：作品 ID、标题/描述原文、作者、发布时间、时长、作品类型、点赞、评论、分享、收藏、话题、封面、分享链接、站内作品链接、视频播放地址、采集时间和收藏合集。分享链接与视频播放地址是必输出字段，不能因为 URL 较长或主表过宽而省略。
4. 可额外保留作者 UID/sec_uid、尺寸、编码格式、播放进度，以及独立的 H264/H265 地址；这些字段放进链接与媒体详情表，避免主表过宽。
5. API 未返回播放量时写 `—`，不要把默认数值 0 表述成真实播放量。
6. 不根据封面或链接推测标题、合集、作者或“为什么收藏”。无法确认的内容写“待确认”。
7. 若目标知识库要求记录“为什么收藏”或“支持什么决策/输出”，在表格中保留这两列；同步阶段不替用户补造，默认写“待确认”，供后续整理。
8. 分享链接按 `share_url`、`share_info.share_url` 等实际响应字段依次提取；视频播放地址优先从 H264 码流的 `play_addr.url_list` 提取，其次使用通用 `video.play_addr.url_list`，再尝试 H265 码流。字段结构以本轮捕获的真实 JSON 为准，不凭空构造媒体域名。
9. 每个 URL 数组至少保存第一个非空完整地址；不得删除查询参数。若存在多个清晰度或编码地址，可用 `<br>` 保留多个 Markdown 链接，并标明 H264/H265 或清晰度。
10. 媒体播放地址可能带时效签名或依赖抖音访问环境。仍需原样保存，并同时记录 `media_captured_at`；不要把“当前可提取”和“永久有效”混为一谈。
11. 某条作品缺少分享链接或视频地址时，不得静默留空或省略列。写 `待确认`，在待确认表和执行报告中记录作品 ID、已检查的候选字段与缺失原因。

字段映射和表格结构见 [Markdown 输出模板](references/markdown-output.md)。

## 6. 读取索引并计算增量

1. 优先使用知识库规则指定的抖音同步索引；其次在目标目录寻找正文明确声明为抖音索引的 Markdown。
2. 索引唯一键为 `aweme_id`，并保存来源 URL、收藏合集、本地表格路径、首次/最近同步时间和状态。
3. 索引重复 ID、路径冲突或机器记录与正文表格不一致时停止写入，列为待确认。
4. 新 ID 标记 `added`；已登记且内容指纹不变标记 `skipped`；来源字段变化标记 `updated`，但只更新 AI 可维护的表格行。
5. 本地标记删除的 ID 保持删除状态并跳过，除非 `restore_missing: true`。
6. 本轮没看到旧 ID 不能证明来源已删除；只有确定性的全量遍历完成后才能标记 `source_state: missing`，仍不得物理删除本地内容。

Python 脚本只维护 `<!-- DOUYIN_SYNC_INDEX:START -->` 至 `<!-- DOUYIN_SYNC_INDEX:END -->` 之间的轻量摘要。完整规范化记录保存在索引旁边的压缩 `.state.json.gz` 状态文件中，避免 Obsidian 打开或渲染数 MB 的 JSON。既有大索引会在首次运行时自动迁移：提取原 `items` 写入压缩状态文件，再把 Markdown 索引缩减为状态文件引用和本次统计。既有索引没有管理标记时会停止，不会覆盖人工文件。

## 7. 写入 Markdown 表格

按知识库规范决定路径。若规范把外部来源统一路由到收件箱，并为抖音定义了来源子目录，则写入该子目录；无规范时使用知识库下的 `douyin-favorites/`。技能中不得硬编码创建者本机的绝对路径、用户名、设备名或 vault 名称。

1. 一个收藏合集对应一个 Markdown 列表文件；未分合集时使用 `抖音收藏列表.md`。
2. 文件包含合规 YAML、同步摘要、收藏主表、必需的链接与媒体地址表、可选技术详情表和待确认项。主表按收藏采集顺序或发布时间倒序稳定排序，并在报告注明排序依据。
3. 标题或描述中的换行替换为空格，竖线转义为 `\|`；链接使用 Markdown 链接，数组用顿号或 `<br>`，避免破坏表格。
4. 表格超过知识库单批上限时分批写入同一文件的多个表格区段或多个分页文件，并在索引中记录全部路径。
5. 只修改明确的 AI 区域；既有文件没有 AI 标记时，另建新的同步列表文件，不覆盖人工内容。
6. 文件和索引均写入成功后才把条目标记为同步成功。
7. 同步只负责采集和入箱，不自动把条目移动到项目、领域、资源、素材或归档目录；需要分类时另行生成建议，等待人工确认。

## 8. 生成执行报告

每次执行都生成本地 Markdown 报告。报告位置优先使用知识库为 AI 衍生内容指定的目录；若未定义，则放在抖音同步目标目录的 `reports/` 子目录。报告至少包括：

| 项目 | 内容 |
|---|---|
| 环境 | 浏览器能力与登录验证结果 |
| 范围 | 收藏合集、时间范围、首屏响应是否捕获、分页完成状态 |
| 统计 | 页面显示总数、当前捕获唯一数、索引原有数、合并唯一数、差额、新增、更新、跳过、失败、待确认 |
