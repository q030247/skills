# 浮墨知识库同步

把 flomo 当作外部源数据，把本地 Markdown 知识库当作目标。先理解目标知识库，再获取浮墨内容；这样可以避免把内容写错目录或破坏本地规范。

## 安全边界

- 只在用户明确要求同步或读取浮墨时访问 flomo。
- 不展示、保存或转发 OAuth 凭据、个人 Token、Cookie 和 Authorization 头。
- 原始浮墨内容按“新增、来源更新、来源删除标记”同步，不擅自物理删除、移动、重新归类或改写来源含义。
- 一条 memo 对应一个 Markdown 文件。来源更新时保留旧版原始内容，再写入新版；本地理解、处理结果及其他人工内容保持不变。
- 先完成笔记正文和附件落盘，再登记为已同步；部分失败不能伪装成成功。
- 批量同步遵守目标知识库的批次上限；没有规定时每批最多 50 篇。
- 涉及个人、公司或其他敏感内容时，只写到用户授权的本地知识库，不调用与同步无关的外部服务。
- 只有用户已经勾选批准的`#候选待办`才允许转换为正式`#task`；勾选表示批准转换，不表示任务完成。
- 候选转换只写入知识库本地文件，不把知识库内容发送给 flomo 或其他外部服务。

## 1. 探测并连接 flomo MCP

1. 查看当前 AI 环境已暴露的 MCP 服务、连接器或工具列表。
2. 不依赖服务器显示名；优先用能力识别。若能看到 `memo_search`，并且服务来源为 flomo，即视为已连接。`memo_create`、`memo_update`可作为辅助证据，但同步主要依赖`memo_search`。
3. 进行最小只读探测。不要为了测试而创建或更新浮墨笔记。
4. 若未连接，优先调用当前平台提供的“添加 MCP/连接器”原生能力，服务器名使用`flomo`，URL 使用`https://flomoapp.com/mcp`，传输协议使用 Streamable HTTP，身份验证优先 OAuth。
5. 若当前 Agent 无权修改平台配置，立即暂停同步，给出该平台最短连接步骤，请用户完成授权后再继续。不要声称已经连接。
6. 若平台不支持 OAuth，再让用户在 flomo 中自行创建 Token，并粘贴到平台的安全凭据界面；不要让用户把 Token 写入对话、Markdown 或项目文件。

常见平台连接方式见[平台连接参考](references/platform-connections.md)。连接属于平台级设置，技能不能假定一次配置会自动在其他 AI 平台生效。

## 2. 定位知识库

按以下证据从强到弱判断当前项目是否指定了知识库：

1. 用户本轮明确指定的文件夹。
2. 项目根目录的`AGENTS.md`、`CLAUDE.md`、`README.md`、`.cursor/rules`或同类说明明确声明的知识库/vault/content 根目录。
3. 当前工作目录本身具有知识库特征，并且规范文件明确把它称为知识库或 vault。

不要仅凭目录中存在若干`.md`文件就认定它是知识库。没有可靠证据时，向用户提供两个选择：①提供已有本地知识库的绝对路径或可解析的相对路径；②将当前 AI 工具所在的工作目录作为知识库。明确展示当前工作目录的解析后路径和选择后的影响，由用户确认其中一种；确认前不读取 flomo 内容，也不新建同步文件。

路径必须位于当前环境允许读写的范围内。若用户给的是相对路径，以项目根目录解析，并在执行前回显最终目标路径。

## 3. 渐进读取知识库规范与模板

先读取索引与规则，不要无目的扫描整个知识库。具体发现顺序见[知识库发现规则](references/knowledge-base-discovery.md)。

1. 读取项目/知识库根目录的 Agent 指令和运行手册。
2. 读取规范文件以及目标目录的 README、主页或索引。
3. 从规则中识别模板目录；若规则未声明，再检查常见目录，如`templates/`、`template/`、`模板/`、`90-模板/`、`.obsidian/templates/`。只读取文件名、frontmatter 和摘要，找到候选后再读正文。
4. 搜索“浮墨”“flomo”“同步索引”以发现既有路由、模板和命名，但限制在规则目录、模板目录、索引和候选目标目录；默认不扫描归档区或正文全集。
5. 把发现的目标目录、必填属性、命名规则、适用模板、索引规则、报告位置、批次上限、安全边界和收件箱生命周期整理成执行参数。
6. 规则冲突时采用优先级：用户本轮明确要求 > 更接近目标文件的指令 > 知识库根规则 > 本技能默认值。模板是格式载体，不能覆盖更高优先级的规范或安全要求。

## 4. 执行兜底决策矩阵

按顺序判断以下情形：

| 知识库 | 规范 | 相关模板 | 处理方式 |
|---|---|---|---|
| 未指定 | 未知 | 未知 | 暂停并让用户二选一：提供本地知识库路径，或确认使用当前 AI 工作目录。展示解析后的路径；确认前不访问 flomo、不创建文件。 |
| 已指定 | 有 | 有 | 规范优先，使用相关模板；模板与规范冲突的字段按规范修正。 |
| 已指定 | 有 | 无 | 按规范生成文件；缺少的排版结构参考技能内置通用模板。 |
| 已指定 | 无 | 有 | 使用相关模板，并补充同步必需的稳定 ID、来源时间和索引字段；不要借机创建整套知识库规范。 |
| 已指定 | 无 | 无 | 使用技能内置的最小目录、笔记模板和索引模板。 |

“相关模板”是指明确适用于外部采集、收件箱、memo、随手记或 flomo 的 Markdown 模板。只因某模板是 Markdown 格式并不代表相关。多个模板都适用且规范无法判定时，列出候选并请用户选择；选择只影响格式时可采用最接近“外部输入/收件箱”的模板并在报告注明。

若知识库规范已有浮墨路由、模板或索引位置，完全遵守。若没有规范也没有相关模板，则在知识库根目录创建`flomo-notes/`，附件放在`flomo-notes/attachments/`，索引使用`flomo-notes/flomo-sync-index.md`，报告使用`flomo-notes/YYYY-MM-DD-flomo-sync-report.md`，笔记使用[通用笔记模板](references/note-template.md)。只创建同步所需文件，不替用户设计整套知识库。

## 5. 读取或创建 Markdown 索引

1. 优先查找规则指定的索引，其次在目标目录查找文件名含`浮墨`或`flomo`且正文声明为同步索引的 Markdown 文件。
2. 找到索引后，先解析 YAML 中的`identity_key`与`items`。若索引损坏、重复 ID 或正文与 YAML 不一致，停止写入并报告待确认，不凭猜测修复历史状态。
3. 没有索引时，按[索引模板](references/index-template.md)创建 Markdown 索引。索引本身也要遵守知识库的必填属性。
4. 身份键优先使用 flomo memo ID，并在本地笔记属性中保存为`source_id`。URL 仅作追溯，不替代稳定 ID。
5. 索引永不删除历史 ID。区分两类删除：本地文件缺失时标记`local_state: missing`；flomo 来源已删除时标记`source_state: deleted`。两者都保留历史记录，不擅自物理删除本地 Markdown。

## 6. 首次运行自动适配知识库

首次运行或知识库规则发生变化时，运行内置发现模块读取根目录`AGENTS.md`、`AI-RUNBOOK.md`和知识库 Schema，并在浮墨来源目录生成`浮墨同步配置.md`。配置至少记录目标目录、附件目录、索引路径、报告目录、批次上限、原文版本策略和收件箱迁移要求。

脚本不直接改写自己的源码。它将规则转换为可复核的本地配置，再按配置改变运行行为；这样既能自动适配不同知识库，也能保留差异、审计和回滚能力。规则文件指纹变化时，先刷新配置，再执行同步。

可以单独预览或生成配置：

```sh
python3 scripts/flomo_sync.py --vault "/absolute/path/to/vault" --configure-only
python3 scripts/flomo_sync.py --vault "/absolute/path/to/vault" --configure-only --apply
```

## 7. 计算增量并同步

优先使用内置脚本`scripts/flomo_sync.py`处理本地差异和写入，避免每次重新实现索引解析、ID 去重和删除标记。MCP 客户端仍负责只读获取数据；不要把 OAuth、Token、Cookie 或请求头交给脚本。

### 脚本输入与调用

1. 用`memo_search`获取候选后，对`content_truncated: true`的条目先用`memo_batch_get`补齐全文。
2. 将完整候选保存为临时 JSON，格式为`{"memos": [...]}`；临时文件放在当前环境允许写入的临时目录，不放凭据。
3. 先运行 dry-run，并审查输出中的`operations`和`counts`：

```sh
python3 scripts/flomo_sync.py \
  --vault "/absolute/path/to/vault" \
  --input "/temporary/path/flomo-memos.json" \
  --target-dir "目标笔记目录" \
  --index "目标笔记目录/浮墨同步索引.md"
```

4. 对照知识库规则核对目录、安全边界、批次和计划；确认无误后增加`--apply`执行。知识库不使用某个历史字段时，脚本不得继续生成该字段。
5. 脚本默认最多处理50条、默认不写文件、拒绝截断正文、拒绝目标路径逃逸知识库。使用`--restore-deleted`前必须有用户明确恢复授权。
6. 脚本生成的是稳定的本地基础稿。若目标知识库存在比脚本默认结构更严格的模板，先使用 dry-run；由 Agent 按目标模板生成笔记后，再用脚本维护索引，或先扩展适配配置，不能绕过本地规范。
7. 人工确认将来源笔记迁出收件箱时，提供`--relocate-manifest`。脚本验证目标文件存在后更新`local_path`、`local_state: moved`、`moved_at`和`destination_type`，不得把迁移误判为删除。

运行参数与输出说明见`scripts/README.md`。

1. 先把索引中的 memo ID、本地路径、来源更新时间和删除状态读入内存。
2. 使用`memo_search`按时间范围获取候选笔记；若工具支持分页，遍历所有页。首次同步且索引为空时，默认拉取从执行时刻向前 3 个自然月的内容；用户明确指定其他范围时以用户要求为准。增量同步从`last_successful_sync_at`适当向前重叠一小段时间，以容纳延迟更新，再依靠 memo ID 去重。
3. 不把空关键词搜索必然等同于“列出全部”；根据工具实际参数使用时间范围、分页或宽范围检索，并记录未覆盖的限制。
4. 对每条候选笔记：
   - ID 未登记：准备新增。
   - ID 已登记且本地标记删除：跳过，除非用户明确要求恢复。
   - ID 已登记且来源更新时间未变：跳过。
   - ID 已登记且来源更新时间较新：把旧原文写入“来源更新历史”，再更新来源属性与“原始内容”章节；保留“它支持什么决策或输出”“我的理解”“处理结果”以及其他人工内容；索引标记`sync_action: updated`并增加`revision_count`。
   - 来源明确返回已删除状态：保留本地 Markdown，索引标记`source_state: deleted`、`sync_action: deleted`、`is_deleted: true`和`deleted_at`。
   - 只有搜索结果缺少某 ID：不能直接判定来源已删除。仅当 MCP 返回删除状态，或当前工具能按相同范围完成确定性的全量枚举与逐 ID 对账时，才标记删除；否则在报告中注明“无法确认来源删除”。
5. 文件名优先使用`YYYY-MM-DD-HHmm-可检索标题.md`。标题来自首个有信息量的文本行，去除标签和非法路径字符，不能确定时使用`浮墨笔记-<memo-id短码>`。
6. 保留原始文字、标签、链接和时间，不把 AI 推断写成原文。正文结构优先来自知识库规范，其次来自相关模板，最后才使用技能内置的通用笔记模板。
7. 若搜索结果含图片或附件 URL，在权限允许时下载到附件目录并改为本地相对引用。下载失败时保留来源信息、将该笔记记为部分失败，且不把附件写成已成功同步。
8. 每条 memo 单独写成一个 Markdown 文件。一个批次准备完成后，把笔记、同步配置、索引和 Markdown 报告先写入临时文件，再统一替换目标文件。
9. 只有报告、笔记和索引全部提交成功，才更新`last_successful_sync_at`；报告失败时水位不得前移。

## 8. 转换已批准候选待办

完成本地规则、候选入口和索引读取后，检查浮墨来源笔记中已勾选的`#候选待办`。候选转换属于每次同步的固定本地收尾步骤；即使本批没有新增 memo，也要执行检查。候选的提出与内容维护仍由`flomo-daily-processing`负责。

仅在以下条件全部满足时转换：

- 候选已勾选；
- 含稳定`candidate_id`；
- 没有`#已转正式任务`；
- 没有`promoted_to`；
- “建议正式位置”唯一、有效且位于当前知识库；
- 与现有正式`#task`不重复。

转换时按以下顺序执行，保证失败时不会留下“已转换”假象：

1. 读取唯一目标文件；需要写入项目主页、领域主页或资源笔记时，只更新既有AI标记区，或新建`<!-- AI:START -->`至`<!-- AI:END -->`区域。
2. 在目标文件创建一条正式任务：`- [ ] #task #领域标签 具体行动`，并附来源双链和验收标准。
3. 确认正式任务已经成功落盘且只存在一份。
4. 在来源候选行添加`#已转正式任务`，并在同一注释中记录`promoted_to`和`promoted_at`；把候选状态更新为“已转正式任务”。

目标不明确、链接失效、疑似重复或目标文件不可写时不要转换，保留原候选并写入报告的“待确认”。已经包含三重幂等信息的候选直接跳过，不重复创建任务。正式任务写入唯一所属项目或领域，`01-驾驶舱/全部待办.md`只通过 Dataview 汇总，不保存副本。

## 9. 生成执行报告

报告至少包括：

- 执行时间、来源、查询范围、目标目录和去重键
- 新增、更新、跳过、删除保持、失败、待确认和附件数量
- 已批准候选数、成功转换数、重复跳过数和转换失败数
- 新增/更新文件链接或路径
- 未覆盖范围、MCP 能力限制、授权问题和失败原因
- 安全检查：是否覆盖、移动、删除或向其他服务发送数据

报告位置和属性优先服从自动适配配置；无规范时使用本技能的通用回退路径。纯跳过批次生成精简报告，出现新增、更新、删除、迁移、失败或待确认时生成完整报告。同步阶段不得擅自搬动来源笔记；人工确认迁移后通过迁移清单完成索引闭环。

## 完成检查

- flomo MCP 的连接状态有真实工具证据，不以猜测代替。
- 目标知识库路径已被明确指定或有可靠项目证据。
- 已读取适用的知识库规范、目标目录说明和候选模板；不存在时已明确采用对应兜底。
- 首次运行已生成本地适配配置；规则指纹变化时已刷新配置。
- 每个新增文件都符合必填属性、命名和目录规范。
- 索引与本地文件逐条对应，ID 唯一，删除状态得到保留。
- 来源更新前的旧原文得到保留，失败项没有登记成成功。
- 已迁移来源记录为`moved`并更新本地路径，没有误记为删除。
- 已批准候选仅在目标唯一且幂等检查通过后转换；正式任务只有一份，来源中已记录`#已转正式任务`、`promoted_to`和`promoted_at`。
- 报告中的数量与实际文件及索引一致；报告失败时索引水位没有前移。
