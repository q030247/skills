# Skills

这是一个面向 AI 智能体的个人技能仓库，提供可复用、可审计的外部信息同步、知识提炼和 Obsidian / Markdown 知识库治理流程。

仓库中的技能以 `SKILL.md` 为入口。它们会先读取目标项目的 Agent 指令、运行手册、Schema 和目录说明，再执行采集、整理或写入；不会假定固定用户名、设备、绝对路径、知识库名称或组织内部结构。

## 推荐工作流

### 浮墨：同步、整理、人工批准、正式任务

```text
flomo 官方 MCP
  → flomo-knowledge-sync
  → 本地原始浮墨 + 同步索引 + 同步报告
  → flomo-daily-processing
  → 闪念卡 + 候选待办 + 文章提炼卡
  → 用户勾选候选，表示批准转换
  → 下一次 flomo-knowledge-sync（即使零新增也检查）
  → 唯一所属项目或领域中的正式 #task
```

职责边界：

- `flomo-daily-processing` 负责理解内容和提出候选，不创建正式任务。
- `flomo-knowledge-sync` 负责同步来源，并转换用户已经批准的候选。
- 候选勾选表示“批准转换”，不表示任务已经完成。
- 正式任务只保存在唯一所属文件；总待办页面应通过查询汇总，不保存副本。
- 浮墨同步技能不再接受迁移清单；来源归档统一交给`close-confirmed-inbox`。

### 飞书妙记：同步、人工审核、提炼、周复盘

```text
飞书妙记
  → feishu-minutes-knowledge-sync
  → 智能纪要 + 原始逐字稿来源包
  → 人工审核逐字稿并放行
  → feishu-minutes-daily-processing
  → 结论、决定、候选待办、业务线索、培训、知识、内容素材、闪念与文章链接
  → 用户勾选候选，表示批准转换
  → 下一次 feishu-minutes-knowledge-sync（即使零新增也检查）
  → 唯一所属项目或领域中的正式 #task
  → feishu-minutes-weekly-review
  → 周度知识增量建议
```

逐字稿没有通过人工审核闸门时，处理技能不得从中提取结论或行动。

飞书与浮墨采用相同的任务职责边界：日处理负责理解内容并生成候选，用户勾选表示批准，对应来源的同步技能负责幂等转换正式任务。飞书原有分类是能力下限，只能增加，不能因统一流程而减少；同一来源允许同时命中多个分类。

### 所有来源：统一归档审批

```text
来源同步或直接写入
  → 收件箱整理与衍生结果
  → close-confirmed-inbox 发现待批准候选
  → 归档审批清单（pending / 待批准）
  → 用户勾选并填写统一审批字段
  → close-confirmed-inbox 执行已批准项
  → 项目材料 / 素材库 / 日志 / 99-归档
  → 状态、双链、来源索引和报告闭环
```

默认`discover_and_execute`模式会先刷新候选，再执行已批准项；也可使用`discover_only`或`execute_only`。候选发现不会自动勾选、批准、填写确认时间或迁移文件。

飞书、浮墨、小红书、抖音、Obsidian直写和其他来源共用相同审批门槛。来源适配器只能增加来源包、附件或索引完整性检查，不能降低门槛。删除不属于无人值守归档授权。

状态字段采用“英文机器值 + 中英展示值”：例如`status: raw`与`status_label: 原始 / Raw`。自动化只判断英文值，新建或触达内容同时维护`中文 / English`展示字段。

## 目录结构

每个技能使用独立文件夹：

```text
skill-name/
├── SKILL.md          # 技能入口、触发条件和完整工作流
├── references/       # 按需读取的模板、规则和操作参考
├── scripts/          # 稳定、重复、容易出错的机械流程
└── evals/            # 测试提示词与预期结果
```

- `SKILL.md` 是唯一入口，frontmatter 中的 `description` 同时决定技能何时触发。
- `references/` 用于渐进加载较长的模板和平台差异。
- `scripts/` 用于固化增量同步、索引维护、审计等确定性操作。
- `evals/` 用于验证正常流程、无增量、重复数据、登录失效和部分失败等边界场景。

## 安装与使用

完整复制所需技能目录到当前 AI 工具支持的技能位置，不要只复制 `SKILL.md`，否则可能缺少脚本和参考文件。

```text
<你的技能目录>/
└── flomo-knowledge-sync/
    ├── SKILL.md
    ├── references/
    ├── scripts/
    └── evals/
```

安装后可以直接描述目标，也可以明确指定技能名称。例如：

- “把今天的浮墨同步到当前 Obsidian 知识库。”
- “整理今天新增的浮墨，提取闪念和候选待办。”
- “我已经勾选了浮墨候选，再同步一次并生成正式任务。”
- “把最近一个月的飞书妙记同步到本地，并保留原始逐字稿。”
- “我已经勾选了飞书候选，再同步一次并生成正式任务。”
- “审计飞书妙记来源包，再整理已经通过人工审核的妙记。”
- “刷新归档审批候选，但先不要迁移。”
- “执行归档审批：先发现候选，再只关闭已经批准且目标明确的来源。”
- “同步我的抖音或小红书收藏到收件箱。”
- “先诊断这个 Obsidian 库，再制定分批整理计划。”

## 设计原则

- **规则优先**：先读取目标知识库或项目的 Agent 指令、运行手册、Schema 和目录说明。
- **渐进读取**：先看索引、属性和摘要，只在任务需要时读取正文。
- **保留源数据**：外部来源和人工记录不被擅自覆盖、改写、移动或删除。
- **稳定去重**：优先使用来源 ID、token 或标准化 URL；重复运行必须保持幂等。
- **人机边界清晰**：AI 负责采集、整理、关联和生成草稿；人负责审核、批准和最终决策。
- **本地适配**：不在技能中固化个人路径、账号、设备、知识库名称或组织内部信息。
- **业务隔离**：不同公司、业务线和敏感来源保持数据边界，只复用抽象方法。
- **可复核**：同步和整理过程生成索引、统计或报告，所有结论可追溯到来源。
- **部分失败可见**：失败项不得登记成成功，报告失败时不得推进同步水位。

## 安全说明

- 不把 Cookie、OAuth Token、访问令牌、密钥或 Authorization 头写入技能、Markdown 或聊天记录。
- 登录、扫码、OAuth 和验证码由用户在官方页面或安全凭据界面完成。
- 删除、批量移动、批量重命名、对外发送和敏感数据上传前必须获得明确确认。
- 涉及合同、财务、证券账户、客户触达和生产系统的行动，只生成分析或草稿，不自动执行。
- 网站接口、CLI 参数和页面结构可能变化；执行时以真实工具证据为准，无法确认时停止并报告。

## 贡献与维护

新增或修改技能时：

1. 保持技能名称和目录名稳定。
2. 在 `description` 中同时说明能力、触发场景和明确边界。
3. 把较长模板或平台差异放入 `references/`。
4. 把稳定且容易出错的机械流程放入 `scripts/`。
5. 为首次运行、零增量、重复数据、来源更新、权限不足和部分失败准备评测。
6. 检查是否包含个人姓名、本机绝对路径、账号、设备或其他不应公开的信息。
7. 修改后验证 JSON、脚本测试、Markdown 链接和打包产物，并保持 README 的技能目录同步。

本仓库会根据实际使用反馈持续迭代。变更应保持通用、可移植和可审计，避免只为单个示例增加硬编码规则。
